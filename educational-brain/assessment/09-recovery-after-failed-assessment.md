# Recovery After Failed Assessment

A failed assessment is a fork in the road, not a verdict. Handled well, it is
the most information-dense event in teaching; handled badly, it is the moment
learners begin to hide. This document is the protocol for handling it well,
every time.

## 1. The failure taxonomy — diagnose before treating

Six causes produce the same observable ("wrong answer"), and they have six
different treatments. Treating one as another ranges from wasteful to harmful.

| # | Cause | Signature | Treatment |
|---|---|---|---|
| 1 | **Item defect** | ambiguity; a defensible alternate reading; multiple learners fail it oddly | discard the result, repair/retire the item — *check this first* |
| 2 | **Affect flood** | performance collapses under assessment framing; same content fine in chat; voice strain | Recovery Engine; assessment over for the session |
| 3 | **Modality/format mismatch** | fails written, passes oral (or reverse); first encounter with the format | re-assess same content, different modality, before concluding anything |
| 4 | **Fragile / unconsolidated** | slow-but-nearly; hedged; knew it yesterday | retrieval reps + interval demotion — NOT re-teaching |
| 5 | **Live misconception** | fast-wrong; systematic; mapped distractor chosen | 7-step repair sequence (Delivery 2 §4) |
| 6 | **Missing prerequisite** | slow-wrong; blank; "I don't know" after effort | prerequisite micro-diagnosis (02 §9) → teach the floor |

## 2. The autopsy — first 60 seconds after a failure

```
FAILURE AUTOPSY
1. SAY NOTHING ALARMED. The tutor's first response sets whether
   failures are safe here. "Interesting — that tells me something
   useful" is both true and load-bearing.
2. NEVER immediately re-ask, and NEVER immediately re-explain.
   Re-asking teaches that wrong answers summon pressure;
   re-explaining before diagnosis treats an unknown disease.
3. CHECK THE ITEM (cause 1): is there a defensible reading under
   which the learner is right? Items are guilty until proven
   innocent — the learner is not.
4. READ THE GRID (causes 4/5/6): fast-wrong → misconception;
   slow-wrong/blank → gap or fragility; hedged-near-miss → fragile.
5. READ THE AFFECT (cause 2): voice flat/clipped/strained,
   posture collapse, "I'm stupid" language → stop diagnosing
   content; the Recovery Engine outranks everything.
6. ONE follow-up, chosen by the grid — a smaller question, a
   "how did you get that?" (said with genuine curiosity, the
   same tone as for a right answer), or the distractor's scripted
   follow-up (03 §8). Not a barrage.
```

The "how did you get that?" discipline deserves its own line: ask it for
**correct answers too**, routinely, or it becomes a tell that announces
wrongness — and the learner starts reversing answers when they hear it
instead of thinking.

## 3. Branch by diagnosis

- **Item defect** → own it out loud ("that was a badly worded question —
  my fault"), discard the data point, log the item for repair. Blaming
  learners for defective items is how instruments stay broken.
- **Affect** → Recovery Engine scripts (Delivery 1). Assessment budget is
  spent for the session regardless of items remaining. End on an engineered
  success (Universal Principle 14). The failed content returns another day,
  restructured.
- **Modality mismatch** → same content, different channel, right now if
  affect allows ("let me ask that a different way"). If it passes: record
  the modality finding (it matters permanently — 06 §4) and count the
  content as passed.
- **Fragile** → this is a *scheduling* event, not a teaching event: demote
  the review interval one step, add immediate retrieval reps (one cued, one
  free), and DO NOT re-teach from zero — re-teaching known-but-rusty
  content is both wasteful and quietly insulting (Delivery 1: cued
  retrieval regenerates faster and repairs confidence).
- **Misconception** → the 7-step repair sequence, *scheduled properly*: if
  the session has affect room, elicit-and-commit now and collide next
  session; never rush collision into a tired or bruised session. Repair is
  surgery — pick the operating conditions.
- **Missing prerequisite** → prerequisite micro-diagnosis flowchart (02 §9)
  → teach the located floor → return to the failed concept LATER via
  isomorph. The learner hears the honest, dignity-preserving version:
  "this one leans on something we haven't made solid yet — that's the
  actual problem, and it's fixable today."

## 4. Re-assessment rules

1. **Never the same item, same session.** By the second retry the learner
   is memorizing the answer, not exercising the skill (01 §5.11).
2. **Isomorph after intervention** — the re-test is always a fresh surface
   of the same structure (03 §4 templates exist for exactly this).
3. **The delay stands.** Re-passing five minutes after remediation proves
   the remediation was *heard*, not that it *held*. The gate's delayed
   component (05 §3) is never waived out of sympathy — waiving it fakes the
   certificate and schedules a worse failure later, on taller scaffolding.
4. **Two failed assessments in one session = circuit breaker.** Assessment
   stops entirely for the session; the session converts to teaching and
   guaranteed-success work. No exceptions — the third failure measures
   nothing (order effects, fatigue) and costs plenty.
5. **Failure streaks across sessions** (same concept, 2+ gates) escalate
   past assessment entirely: the teaching path itself is defective for
   this learner — different representation family, different action family
   (Delivery 2 §6 rotation), possibly a mis-mapped prerequisite chain.
   Log it as lesson-design feedback; the moat grows at exactly these points.

## 5. What the learner hears

Language discipline, absolute:

- The Brain never says "you failed", "wrong again", or "let's try an easier
  one" (audible demotion). It says **"not yet"**, names the specific found
  cause, and names the specific next step: "we found the exact missing
  piece — that's what these checks are for."
- Failures are *the check working*: "the check did its job" reframes the
  instrument as being on the learner's side — which it is.
- Results are private by default; progress is reported against the
  learner's own past, never against others (07 §4).
- The learner is never told a number that means nothing to them ("you're
  at 62%"); they're told what they can now do and what comes next.

## 6. The full recovery flowchart

```
ASSESSMENT ITEM FAILED
│
├─ 0. Tone: neutral-warm. No alarm, no re-ask, no re-explain.
│
├─ 1. Item defensible-reading check ──defect──► discard result,
│                                               repair item, own it aloud
├─ 2. Affect check (voice, language, posture)
│        └─ flooded ──► RECOVERY ENGINE; assessment closed for
│                       session; end on success; content returns
│                       another day
├─ 3. Latency × correctness grid
│        ├─ fast-wrong ─────────► MISCONCEPTION path:
│        │                        elicit → commit now; collide when
│        │                        conditions are right; full 7-step;
│        │                        delayed disguised re-probe later
│        ├─ hedged near-miss ───► FRAGILE path:
│        │                        cued retrieval now; interval −1;
│        │                        no re-teaching
│        └─ slow-wrong / blank ─► 4.
│
├─ 4. Modality cross-check ("let me ask another way")
│        └─ passes ──► record modality finding; count as passed
│
├─ 5. Prerequisite micro-diagnosis (02 §9)
│        ├─ floor found ──► teach floor → isomorph on the failed
│        │                  concept LATER (not same-session)
│        └─ floor solid ──► the concept itself needs a changed
│                           representation (three-representation
│                           rule) — teaching problem, not learner
│
├─ 6. Session bookkeeping: visible-failure count +1
│        └─ count = 2 ──► CIRCUIT BREAKER: no more assessment
│                         this session; teach; end on success
│
└─ 7. Always: schedule the delayed re-verification; log the
     failure cause against the item AND the lesson design
     (repeated cause = the moat's next authoring target).
```
