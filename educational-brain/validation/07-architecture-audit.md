# Architecture Audit — Educational Brain vs. Runtime Reality

**Date**: 2026-07-10  
**Method**: Read every authored Brain document (Deliveries 1–10) and every
relevant runtime file (`route.ts`, `teaching-engine/index.ts`,
`curriculum/route.ts`, `onboarding/route.ts`, `teachingStrategy.ts`,
`getDashboardV2Data.ts`, `placement.ts`) from the live Git tree, then
compare what the Brain says should happen against what the code does.

This document describes **actual behavior** only. It does not describe
ideal behavior, planned behavior, or what the ADRs propose. Where the
runtime contradicts the Brain, the contradiction is named plainly.

---

## Part 1 — Complete Runtime Trace

### "I already know some mathematics" — what actually happens

A Library mode learner types this in the chat. Second by second:

**1. Message ingestion** (`/api/learn/chat` POST handler):  
The message joins the conversation history. History is capped at 30
messages. No parser intercepts the claim. No trust model fires. No
placement protocol launches. The text is treated as a natural-language
utterance, nothing more.

**2. System prompt assembly** (~25 blocks, each try/catch-wrapped):  
`buildTutorSystemPrompt()` runs first with:
- `profile.selfDescription` (free text from onboarding, raw)
- `profile.currentLevel` (beginner | intermediate | advanced — 3 tiers)
- `lessonCtx`: current lesson number and KG node title from
  `StudentProgress.currentLesson` (or `computeCurriculumEntryOrder()`
  if no real progress row exists yet)
- `memoryContext`: the JSONB `contextSnapshot` from the previous turn

Then the remaining ~24 blocks append unconditionally. For Library mode,
the operative blocks are:
- KG context block: node title, description, prerequisites,
  estimated hours, WHY THIS TOPIC MATTERS
- Teaching strategy block: `getTeachingStrategy()` → `determineStrategy()`
  → one of 7 type labels → `buildTeachingStrategyBlock()` → 5 hardcoded
  instruction strings + 1 hardcoded action directive
- Misconception block (flag-gated): generic alert if MistakeRecord rows
  exist, confidence tier only
- Learner Intelligence Profile: aggregated TopicProgress + MistakeRecord
  + EvidenceRecord summary
- Spaced revision block: due review items from retentionMetrics where
  score < 70 or review due ≤ 7 days
- Lesson plan block: the `LessonPlan` JSON stored in contextSnapshot,
  which was generated by an LLM at session start (not from authored rules)
- Visual aids block

The Teaching Engine decision block — `decide()` from
`teaching-engine/index.ts` — **does not fire** for Library mode unless
`ENABLE_LIBRARY_CONCEPT_TRACKING=1` is set (off by default). The Teaching
Action Generator (`getTeachingAction`) and Dynamic Lesson Composer
(`getLessonPlan`) run only when `schoolCtx` is set. Neither is present for
Library mode.

**3. No authored placement protocol runs.**  
`assessment/02-diagnostic-assessment.md`'s binary-search protocol, the
trust model from `placement/02-level-hypothesis.md`, the three-bracket
verification, the affect budget limit of two visible failures — none of
these are wired to any runtime path that fires in response to "I already
know some mathematics." The claim vanishes into the 30-message history.

**4. `routeAI()` is called:**
```
routeAI(messages, systemPrompt, country, 1024, teachingLang, ...)
```
Groq (`openai/gpt-oss-20b`) receives the system prompt + 30-message
history + the new message. Max output tokens: 1024 (hardcoded).

**5. The AI invents its response.**  
The AI sees the assembled context and generates a reply. If it happens
to ask a diagnostic question about prior mathematics knowledge, it does
so through its own LLM reasoning. No authored protocol governs this. The
response is not constrained by the binary-search placement algorithm, the
fluency-judged probe format, or the never-reveal-the-discrepancy rule from
`placement/02 §4`.

**6. Post-processing and persistence:**  
Response text is parsed for tags (`VISUAL:`, `WE:`, `INLINE_PRACTICE`,
`HINT`). If `isExplanationMemoryEnabled()` and `resolvedConceptId` is set,
`assembleLesson()` was attempted BEFORE the LLM call — but it returns
`null` today because no assets have been promoted to ACTIVE status in the
AssetIdentity catalogue. `ingestGeneratedLesson()` runs fire-and-forget
to save the LLM output as DRAFT. `appendEvidenceEvent()` fires
fire-and-forget. `Message` row saved. `contextSnapshot` JSONB updated.

**What is actually retrieved from the Educational Brain at runtime:**  
Nothing. Zero. No runtime path reads any file in `educational-brain/`.
The tree is unreachable from the teaching pipeline. The authored knowledge
exists only as documents — the retrieval mechanism exists architecturally
(AssetIdentity DB tables, admin approval flow, `assembleLesson()`) but
carries no authored content yet.

**What the runtime does retrieve from structured sources:**  
TopicProgress rows (concept-level mastery floats), MistakeRecord rows
(misconception flags with confidence), EvidenceRecord rows (ADR 13
evidence events), TeachingStrategyEvent rows (stalemate detection),
contextSnapshot JSONB (previous concept, lesson stage, worked example
state), retentionMetrics (spaced repetition schedule), KG graph JSON
files (node metadata, prerequisites, unlocks).

## Part 2 — AI Reasoning Remaining (Ranked by Impact)

Every item here is knowledge the AI invents every turn that could be
retrieved from the authored Brain. Ranked by how many turns the gap
affects and how much authored content already exists to fill it.

### 1. Teaching action selection — 100% of Library mode turns

`decision-engine/04-action-selector.md` authors a 7-filter funnel
producing one of 27 action types. The 27-action dispatch library (Delivery
2 §6), the persona veto table, the cognitive-load cost table — all
authored, none available to the runtime.

In Library mode, the AI receives 5 hardcoded strings from
`STRATEGY_INSTRUCTIONS[strategyType]` in `teachingStrategy.ts` and
invents the actual teaching action. The same 5 hardcoded strings appear
regardless of which concept is being taught, who the learner is, what
rung they are on, or what misconceptions are active.

**Authored content waiting:** 27 actions with dispatch criteria, the
7-filter funnel, the rung → action-legality table. This is the single
most consequential gap.

### 2. Prior-knowledge placement — every new or returning learner

`assessment/02-diagnostic-assessment.md` + `placement/` (Delivery 10)
author the full binary-search placement protocol: trust model by
age/context, three-bracket verification with fluency judgment, affect-
budget cap of 2 visible failures, binary-search spine, patchy-history
flag, per-branch frontier.

In the runtime: `computeCurriculumEntryOrder(graph, level)` finds the
first KG node at or above a difficulty floor (foundational/proficient/
advanced for beginner/intermediate/advanced). This is the simplest
possible proxy. It:
- Ignores branch asymmetry (Student E simulation)
- Ignores the trust model (Student F simulation)
- Ignores the never-reveal rule (Student B simulation)
- Places a learner at a node whose prerequisites may be unmet
- Does not probe; does not adjust on evidence

The claim "I already know some mathematics" changes nothing in the
runtime's placement. The authored placement protocol would change
everything.

### 3. Misconception detection and repair — every misconception event

`detectMisconceptions()` reads MistakeRecord rows and generates a generic
alert: concept slug, confidence level, count. The authored misconception
library in `concepts/math.arith.fractions.md` has: verbatim
characteristic phrases ("I just add top and bottom"), specific elicit-
commit-collide sequences, burned-collision tracking, regrowth priors,
repair-path redesign triggers.

The AI receives: "ACTIVE MISCONCEPTION: fraction equivalence, confidence
HIGH." It invents the repair. The authored collision sequence, verbatim
probe design, the regrowth-rate re-rating rule if this is the second
repair failure — all invented.

### 4. First-lesson compliance — every complete beginner at lesson one

`first-lesson/` (Delivery 6): 8 documents. Never-rules (1 concept, ≤3
new words × 3 uses each, 2-sentence bursts, ≤6 questions, 1-failure
budget), the demonstrate-before-explain sequence, the C1–C4 completion
criteria, the 16-entry AI anti-library, subject adaptations anchored to
verified KG entry nodes (English lesson one is ORAL, print-optional).

In the runtime: a Library mode beginner starting `eng.phonics.letter-
sound-correspondence` receives the same system prompt structure as an
intermediate student resuming mid-curriculum. The currentLevel field
changes the difficulty floor for placement and the contentRegister label
in the base prompt ("beginner" | "intermediate" | "expert"). No
deterministic filter enforces the 1-concept limit, the ≤6-question
cap, or the demonstrate-before-explain sequence. These rules are not in
the system prompt. Whether the AI follows them depends on its training.

### 5. Explanation selection — every turn with new content

ADR 14's AssetIdentity path (`assembleLesson()`) is the runtime retrieval
mechanism for authored explanations. It is wired into `route.ts` and
fires before the LLM call when `resolvedConceptId` is set. It always
returns `null` today because the ACTIVE catalogue is empty.

The authored explanation libraries in `concepts/` have: primary
explanation, alternative representations (concrete/visual/abstract),
anti-analogies (analogies that are specifically wrong for this concept
and must be avoided), demonstration sequences. These exist in files that
no runtime path reads.

### 6. Session shape (OPENING/CORE/CLOSE) — every session

`decision-engine/07-lesson-planning-engine.md` authors the invariant
session shape: OPENING (due reviews, warm-up), CORE (one frontier
concept, fluency-gated triad), CLOSE (summit-ending, visible horizon).

In Library mode, the lesson plan block is generated by the LLM at session
start and stored in contextSnapshot. It may or may not follow the
invariant shape. There is no deterministic enforcement of OPENING-before-
CORE, no compaction trigger, no stuck-concept protocol (pivot to
prerequisite repair after N failed sessions on the same concept).

The stalemate detection in `getStrategyEffectiveness()` detects a stale
strategy and excludes it. But it does NOT route to the prerequisite repair
track — it selects the next-priority strategy for the same concept.

### 7. Teaching state machine — every turn

`decision-engine/01-teaching-state-machine.md` authors 10 named teaching
states with explicit transition rules and a RECOVERY preemption rule
(fires before any other transition; exits one rung below entry state).

In the runtime: there is no explicit state machine. Teaching state is
implicit in the combination of `TeachingStrategyType`, `masteryLevel`,
and conversation history. The AI infers current state from context.
RECOVERY does not preempt — it is one of the 7 strategy types, selected
by priority ordering when momentum is DISENGAGEMENT_RISK.

### 8. Knowledge-ladder rung tracking — every mastery assessment

`student-state/02-knowledge-state.md` authors 8 rungs (UNKNOWN through
EXPERT) with specific evidence requirements for each transition:
INDEPENDENT requires correct with minimal prompting; AUTOMATIC requires
correct, fast, no errors; TRANSFER requires correct application to novel
surface.

In the runtime: `masteryScore` is a 0–100 float accumulated from
checkpoint evaluations. `MasteryLevel` is AT_RISK / FALSE_MASTERY /
DEVELOPING / TRUE_MASTERY. TRUE_MASTERY maps roughly to AUTOMATIC on the
knowledge ladder, but does not enforce the STABLE condition (21-day spaced
retrieval pass) or the transfer-event requirement. The never-reteach law
fires when `masteryLevel === 'TRUE_MASTERY'` selects ACCELERATED_
ADVANCEMENT — but TRUE_MASTERY can be reached faster than ANCHORED because
the evidence bar is lower.

## Part 3 — Human Tutor Micro-Decision Audit

### English: Letter-Sound Correspondence

A great human tutor's first 10 decisions for lesson one:
1. Distinguish letter name ("ay") from letter sound (/æ/) — they are
   different, not interchangeable. Never combine them in one utterance.
2. Check phonological awareness before letter-sound work: can this learner
   isolate a spoken sound at all?
3. Demonstrate first: "Listen to me say 'apple' — /æ/, apple. What sound
   does 'apple' start with?" — not "The letter A makes the sound /æ/."
4. Echo before solo: "Say /æ/" before "What does A sound like?"
5. Keep the lesson ORAL. Print is optional at lesson one.
6. One sound per lesson. Not five.
7. The word must be familiar to the learner (not a vocabulary lesson
   disguised as a phonics lesson).
8. If the learner mishears, redirect without marking wrong: "Listen again
   — /æ/ — apple. Now you."
9. Close when C1 is met (one correct solo response), not when the tutor
   finishes explaining.
10. Lesson one completion criterion is lesson two's OPENING retrieval —
    never test at lesson-one close.

**What actually happens:**  
`computeCurriculumEntryOrder(graph, 'beginner')` places the learner at
the first foundational node — `eng.phonics.print-concepts` or
`eng.phonics.phonemic-awareness` depending on the KG structure. The
system prompt receives the node title and description from the KG JSON.
The teaching strategy is selected (probably FOUNDATION_BUILDING) and the
AI receives 5 hardcoded instructions: "Start from first principles — do
not assume any prior knowledge." "Use simple, encouraging language."
Etc. The AI generates its own lesson one content from this.

The authored `first-lesson/` library: 1-concept limit, ≤3 new words, 2-
slot WM treatment, demonstrate-before-explain, echo-before-solo, C1–C4
completion criteria, 16-entry AI anti-library (quiz-first anti-pattern,
definition-first anti-pattern, echo-advancement anti-pattern, register
drift anti-pattern), ORAL-first subject adaptation for English — none of
this is in the system prompt. The AI may or may not violate these rules.
There is no enforcement.

**Gap severity**: CRITICAL. First-lesson failure was the documented
platform failure that motivated Delivery 6. The authored fix is
complete. The fix is not wired.

### Mathematics: Fractions (M1 active: add-across misconception)

A great human tutor's micro-decisions when M1 is active:
1. Elicit first — never correct before committing: "Quick — what's
   1/2 + 1/3?"
2. Get an explicit commitment: "So you're saying 2/5. OK. Let's check."
3. Create cognitive conflict with a concrete anchor: "If you eat half
   a pizza and then a third of a pizza, did you really only eat 2/5 —
   less than half?"
4. The characteristic phrase "I just add top and bottom" is diagnostic —
   the moment it appears, route to the M1 repair track
5. The collision evidence must be PHYSICAL or VISUAL — verbal argument
   alone does not dislodge M1 (the misconception lives at the procedural
   level, not the propositional level)
6. Do not teach the LCD algorithm until the concept of equivalent fractions
   is grounded
7. After repair: one transfer-distanced probe — not the same surface

**What actually happens:**  
`detectMisconceptions()` reads MistakeRecord rows. If M1 has been flagged,
the system prompt receives: "ACTIVE MISCONCEPTION: fraction equivalence,
confidence: HIGH, count: 3." Teaching strategy: MISCONCEPTION_CORRECTION.
Hardcoded instructions include: "Address the specific misconception before
advancing." "Use Socratic questioning to surface the gap." The AI invents
the collision sequence.

The authored M1 content in `concepts/math.arith.fractions.md`: verbatim
diagnostic phrase ("I just add top and bottom"), specific collision
sequence (pizza-sharing anchor → counting fractional parts → why
denominator reflects the size of each part), anti-analogy warning (do not
use the number-line analogy for M1 repair — it activates a different
representation that leaves M1 intact), repair confirmation probe design,
burned-collision tracking (if this collision failed before, try the
physical splitting representation instead) — NONE of this is retrieved.
The AI invents a generic misconception-correction lesson.

**Gap severity**: HIGH. M1 (add-across) is among the most stubborn
fraction misconceptions. The authored repair sequence is correct and
specific. The runtime substitutes generic Socratic prompting.

### Physics: Newton's First Law (bored-advanced profile)

A great human tutor's micro-decisions when the learner is bored-advanced:
1. Recognize the signal: one-word answers, instant correct responses, no
   visible processing — this is not mastery display, this is disengagement
2. Switch to OBSERVATION stance immediately — stop explaining, start
   probing
3. The transfer challenge is the diagnostic: "A hockey puck slides on a
   perfectly frictionless ice surface. What happens to it?"
4. If correct: "Now — the puck is moving. You push it in the same
   direction it's already moving. What changes?" (distinguishing
   force-with-motion from force-causing-motion)
5. The impetus misconception (the learner's internal force model) must be
   surfaced even when the learner gives correct answers — correct answers
   without the mechanism are a FALSE_MASTERY signal
6. Generative mode: the learner poses the extension challenge, not the
   tutor (metacognitive engagement)

**What actually happens:**  
`getLearningMomentum()` detects STRONG_MOMENTUM. `getMasteryProfile()`
returns TRUE_MASTERY if scores are high. Strategy: ACCELERATED_ADVANCEMENT.
Hardcoded instructions: "Move quickly to advanced applications." "Skip
foundational reviews." "Challenge with complex multi-step problems." The
AI invents a set of challenging questions about Newton's First Law.

The authored content in `concepts/phys.mech.newtons-first-law.md`: the
impetus misconception (pre-Newtonian force model that survives even after
learners learn the correct statement), specific transfer scenarios
(astronaut in space, object thrown horizontally vs. dropped, elevator
acceleration), the anti-analogy list (do not use a ball rolling to a stop
as a "no force" example — friction is present; it reinforces the wrong
model), the specific transfer-level probe design — NONE retrieved. The AI
produces transfer questions from its own training knowledge about
Newton's First Law, which may or may not match the concept entry's
authored probe sequence.

**Gap severity**: MEDIUM-HIGH. The impetus misconception is the most
common advanced misconception in Newtonian mechanics and the hardest to
surface. The authored probes are specifically designed to surface it. The
AI may miss it entirely.

## Part 4 — Orchestration Audit

**Can the authored libraries work together without a new orchestrator?**

Short answer: Yes — the orchestrator exists. `route.ts` is the
orchestrator. It assembles ~25 blocks and calls the LLM. The problem is
not that the orchestrator is missing; the problem is that the content it
retrieves does not come from the authored Brain libraries.

**Current retrieval sources for the orchestrator:**
1. PostgreSQL database (user data, progress, misconceptions, evidence)
2. Hardcoded TypeScript constants (strategy instructions, action directives)
3. KG JSON files (node metadata, prerequisites, unlocks)
4. Hardcoded configuration (HISTORY_LIMIT=30, MAX_TOKENS=1024)
5. ADR 14 AssetIdentity DB tables — wired but empty

**What the authored Brain has that the orchestrator never reads:**
- `educational-brain/` text files — entirely unreachable at runtime
- The 7-filter action selector logic (`decision-engine/04`)
- The 10-state teaching state machine (`decision-engine/01`)
- The invariant session shape rules (`decision-engine/07`)
- The first-lesson constraints (`first-lesson/`)
- The binary-search placement protocol (`assessment/02`, `placement/`)
- Concept-specific misconception repair sequences (`concepts/`)
- The 8-rung knowledge ladder evidence requirements (`student-state/02`)
- The warm-up vs. reteach distinction for returning learners (`placement/06`)

**The actual disconnection mechanism:**  
The Brain is a document library. The orchestrator reads structured data
(DB, JSON, constants). The designed bridge between them is the ADR 14
AssetIdentity pipeline: author content in Brain → populate DB tables via
DRAFT → admin review → ACTIVE → `assembleLesson()` serves it to the
orchestrator before the LLM call.

This bridge is complete for the EXPLANATION and PROBE asset families. It
is wired. It has never carried content. The pipeline exists; the content
has not been imported.

**What the bridge cannot carry (structural gap):**  
The AssetIdentity pipeline serves stored assets (explanations, probes,
visuals). It cannot serve decision rules (when to use which explanation,
how to transition between teaching states, when to preempt with RECOVERY).
Decision rules are control-flow logic, not content. They need to become
code, not DB rows.

The authored rules in `decision-engine/` and `placement/` need to be
translated into runtime logic — not into the LLM prompt, into the
orchestrator's logic before the prompt assembly. The first-lesson never-
rules are an example: they are not LLM instructions, they are constraints
on what the LLM is allowed to receive.

**Conclusion:** No new orchestration layer is needed. Route.ts is the
orchestrator. What is needed:
1. Populate the AssetIdentity pipeline with authored concept entries
2. Translate the decision-engine rules into deterministic pre-LLM filters
3. Wire the placement protocol into the Library mode session start
4. Enforce the session shape deterministically for Library mode (not via
   LLM-generated lesson plan)

## Part 5 — Reorganization vs. New Library

**The case for reorganization:**

Every gap in Part 2 corresponds to authored content that already exists in
the Brain and a runtime location that already accepts structured input.
There is no gap that requires a new document library. There are gaps that
require the existing library to be connected.

**The map:**

| Gap | Authored knowledge location | Runtime connection point |
|---|---|---|
| Action selection | `decision-engine/04`, D2 §6 | `getTeachingAction()` (TAI) — extend to Library mode |
| Placement protocol | `assessment/02`, `placement/03` | `placement.ts` — replace difficulty-floor with binary-search |
| Misconception repair | `concepts/` entries | AssetIdentity PROBE → `assembleLesson()` |
| First-lesson constraints | `first-lesson/` | Deterministic filter in route.ts before prompt assembly |
| Explanation content | `concepts/` entries | AssetIdentity EXPLANATION → `assembleLesson()` |
| Session shape | `decision-engine/07` | Session init logic for Library mode (not LLM-generated) |
| Teaching state machine | `decision-engine/01` | State tracker in contextSnapshot (currently absent) |
| Knowledge-ladder rungs | `student-state/02` | `ConceptMasteryRecord` rung field (ADR 10, not yet implemented) |

**The one potential new structure:**  
The authored decision rules in `decision-engine/` cannot be served as
AssetIdentity rows — they are control-flow logic, not content. If
they are going to be retrieved rather than hardcoded, they need either:
(a) to be translated into TypeScript (which is implementation, requires
G2 approval), or
(b) a structured rule-retrieval format (a new type of DB row: RuleAsset
or PolicyAsset, distinct from EXPLANATION/PROBE/VISUAL).

This is the only gap where a new structure might be warranted. It is NOT
a new educational library — it is a new asset family in the existing
AssetIdentity model (ADR 14 §8 already defines VISUAL as a third family;
INSTRUCTION/POLICY would be a fourth). This is an extension of existing
architecture, not a new parallel system.

## Part 6 — Unnecessary Reteaching

**Where the runtime reteaches unnecessarily:**

### 1. Returning learner invisible restart

`placement/06-resumption.md` authors the warm-up protocol for returning
learners: cued retrieval, not re-introduction; cascade-unlock expectation;
re-placement from high-water mark (not from beginning) for gaps >6 months.

In the runtime: when a learner returns after any gap, the session
continues from `StudentProgress.currentLesson`. There is no gap-length
detection. There is no warm-up protocol. There is no distinction between
a 2-day gap and a 14-month gap. For a 14-month returning learner:
- `currentLesson` is at the position before they left
- The lesson content is that frontier concept
- No OPENING warm-up of the categories below the frontier
- The LLM may or may not introduce a warm-up question based on the
  conversation history it receives

This is the invisible restart anti-pattern documented in
`validation/02-failure-replay.md` (Failure 2). It was identified as fully
preventable by the authored knowledge. It remains unprevented.

### 2. Mastered concept reteaching

The never-reteach law (`placement/04 §1`): "Never enter a mastered
category as a teaching target."

In the runtime: if `masteryLevel === 'TRUE_MASTERY'`, strategy
ACCELERATED_ADVANCEMENT fires and the LLM is instructed to "Move quickly
to applications." This is close to the never-reteach law but not
identical:
- TRUE_MASTERY can be declared without the STABLE condition (21-day
  spaced retrieval pass)
- TRUE_MASTERY can be declared without a transfer event
- ACCELERATED_ADVANCEMENT still enters the concept — it does not skip it
  to the next frontier

A learner whose concept is ANCHORED in the Brain's framework should never
receive CORE teaching on that concept. In the runtime, they may receive
CORE teaching under ACCELERATED_ADVANCEMENT if the lesson plan routes to
it.

### 3. Stalemate resolution reteaches the same concept

`decision-engine/07 §7` (stuck-concept protocol): if the learner is stuck
for N sessions, pivot to prerequisite repair — not more of the same
concept.

In the runtime: `getStrategyEffectiveness()` detects stalemate and passes
`excludeStrategy` to `determineStrategy()`. The next strategy in priority
order is selected. This is likely to be MISCONCEPTION_CORRECTION or
STEP_BY_STEP_GUIDED — still on the same concept. There is no rule that
routes to the prerequisite repair track. The learner who is stuck on
fraction equivalence receives progressively different teaching approaches
for fraction equivalence, not a prerequisite repair of counting and
multiplication (which the authored protocol would prescribe).

## Part 7 — Architect Review

### Finding 1: The Educational Brain and the runtime are parallel universes

Every document in `educational-brain/` is unreachable at runtime. The
tree is the right kind of knowledge. The production pipeline that reads it
does not exist. This is not a design flaw — it was the explicit plan (the
charter says "authored once for permanent retrieval," ADR 14 designs the
retrieval mechanism). But the actual state is: 52 authored retrievable
layers, 0 retrieved.

The charter's long-term objective ("reduce AI reasoning over time") is
currently at its starting position: AI invents everything, every turn,
from the LLM's training weights plus the generic context in the system
prompt.

### Finding 2: `route.ts` is the real educational Brain

The 2069-line `route.ts` is where every teaching decision actually
executes. It assembles ~25 context blocks unconditionally, calls the LLM,
and parses the result. The LLM resolves conflicts between the blocks (which
instruction to follow when the misconception block says "address the
misconception" and the momentum block says "keep the energy high and move
fast"). This LLM-as-conflict-resolver is the undocumented behavior of
the system. It is not in any ADR. It is not in any Brain document.

The Session Recommendation Reconciler (flag-gated) covers 4 of the ~25
signals with a deterministic priority table. The other 21 signals are
resolved by the LLM implicitly. This is the deepest form of "AI reasoning
remaining" — not about what to teach, but about which instructions to
follow about how to teach.

### Finding 3: The Teaching Engine is school-only in practice

`decide()` (teaching-engine/index.ts) requires `ENABLE_LIBRARY_CONCEPT_
TRACKING=1` for Library mode. Off by default. `getTeachingAction()` and
`getLessonPlan()` require `schoolCtx`. The documented architecture (ADR
08, ADR 09) acknowledges this gap and proposes Library mode extensions.
The gap has not closed. Library mode learners — the primary use case for
the subject-KG platform — receive a significantly stripped-down experience
versus school mode.

### Finding 4: The strategy layer and the action layer are disconnected

`teachingStrategy.ts` produces a 7-type strategy (macro level). `teaching-
engine/index.ts` produces a 4-type ActionType (micro level). The Brain
authors two distinct layers: the session-level teaching state (10 states
in `decision-engine/01`) and the turn-level action (27 types in the
pending D2 §6). In the runtime, the macro layer (strategy) runs for both
modes; the micro layer (action) runs only for school mode; the 27-action
dispatch library does not exist in code at all. The action selector
(`decision-engine/04`) is authored with 7 filters but the libraries its
filters 2–6 reference (knowledge type, state-legality table, persona veto
table, load cost table, affinities) are all pending Deliveries 1–2
transcription.

### Finding 5: `placement.ts` is a correct minimum but not the authored protocol

`computeCurriculumEntryOrder(graph, level)` correctly moves a learner past
difficulty-appropriate content at first session. It does not:
- Probe to verify the learner belongs at that level
- Adjust based on the three-bracket verification results
- Apply the trust model to the self-reported level
- Handle branch asymmetry
- Detect patchy-history
- Enforce the never-reteach law on individual concepts before the entry
  point

These are `placement/01–06` authored content. The implementation is a
1-dimensional approximation (difficulty floor) of an 8-phase protocol.

### Finding 6: The onboarding data model is insufficient for the authored placement protocol

Onboarding collects: level (3 tiers), selfDescription (free text, raw),
subjects, teachingLanguage, voiceChoice. No structured prior-knowledge
data. No misconception probing. No prerequisite mapping. The full
placement protocol requires: trust-calibrated self-report, three-bracket
verification, per-branch frontier, patchy-history detection. None of this
is triggered at or after onboarding. The first session starts with the
placement approximation and never refines it through the authored protocol.

### Finding 7: The AssetIdentity pipeline is the right architecture, the wrong activation state

The ADR 14 implementation (`explanationMemory.ts`, `teachingActionRepository.ts`,
`matcher.ts`, `/api/admin/knowledge-assets`) is architecturally correct.
The pipeline is wired into `route.ts`. The student state builder
(`studentState.ts`) is correct. The admin approval flow is correct.

The activation gap: the authored concept entries in `educational-brain/
concepts/` have not been imported as DRAFT assets. The admin review queue
is empty. Nothing has been promoted to ACTIVE. `assembleLesson()` always
returns `null`. The LLM path runs unchanged from before ADR 14 was built.

The authored concept entries for three subjects
(`math.arith.fractions`, `phys.mech.newtons-first-law`,
`eng.phonics.letter-sound-correspondence`) are exactly the format needed
for AssetIdentity EXPLANATION and PROBE assets. The data exists; the
import step hasn't run.

### Finding 8: What "reducing AI reasoning" requires in practice

The charter says: "reduce AI reasoning over time by authoring educational
knowledge once for permanent retrieval." What this requires in practice:

**For content (explanations, probes, examples):** The AssetIdentity
pipeline is the mechanism. It requires: (a) importing authored concept
entries as DRAFT assets, (b) admin review to ACTIVE, (c) `assembleLesson()`
then serves them — the LLM never generates what is already ACTIVE.

**For decision rules (when to use which explanation, how to transition
states, when to preempt with RECOVERY, when to launch the placement
protocol):** The AssetIdentity pipeline cannot carry these. They are
control-flow logic that must become deterministic code before the system
prompt is assembled. A POLICY/INSTRUCTION asset family (extension of ADR
14) or direct code translation of the authored rules is required.

**The fundamental limit:** No amount of authored content in the Brain tree
reduces AI reasoning at runtime unless that content is connected to a
retrieval path. The connection paths that exist today:
- AssetIdentity EXPLANATION + PROBE → `assembleLesson()` (wired, empty)
- KG JSON → system prompt KG context block (wired, flowing)
- DB mastery/misconception/memory data → system prompt blocks (wired, flowing)

The connection paths that do not exist:
- `educational-brain/` decision rules → deterministic pre-LLM filters
- `educational-brain/first-lesson/` constraints → system prompt enforcement
- `educational-brain/placement/` protocol → Library mode session init
- `educational-brain/concepts/` entries → AssetIdentity DRAFT import

## Gap Summary

| Gap | Impact scope | Authored content ready | Runtime connection exists |
|---|---|---|---|
| Teaching action selection | Every Library turn | D2 §6 pending transcription | Partial (TAI in school mode only) |
| Placement protocol | Every new/returning learner | `assessment/02`, `placement/01–06` | `placement.ts` (difficulty floor proxy only) |

| Explanation content retrieval | Every turn with new content | `concepts/` entries | AssetIdentity EXPLANATION (wired, empty) |
| Session shape determinism | Every Library session | `decision-engine/07` | None (LLM-generated lesson plan) |
| Teaching state machine | Every state transition | `decision-engine/01` | None (implicit in strategy type) |
| Knowledge-ladder rung tracking | Every mastery event | `student-state/02` | No rung field in runtime (ADR 10 pending) |
| Warm-up for returning learners | Every gap >3 days | `placement/06` | None |
| Decision-rule retrieval | All of the above | Decision-engine docs | None (rules not in AssetIdentity model) |

**Priority order for closing these gaps (architecture only):**

1. Import authored concept entries into AssetIdentity DRAFT and begin
   admin review for EXPLANATION and PROBE assets — this activates an
   existing, wired pipeline and immediately reduces AI invention for
   the three authored concepts
2. Add POLICY/INSTRUCTION as a fourth AssetIdentity family — this extends
   the existing architecture to carry decision rules, not just content
3. Translate the first-lesson constraints into a deterministic pre-LLM
   filter in `route.ts` — this is the highest-impact, simplest-to-
   implement rule enforcement
4. Replace the difficulty-floor placement with the binary-search protocol
   from `assessment/02` for Library mode session starts
5. Extend `decide()` to fire for Library mode (the flag path already
   exists; the concept-ID write path needs to be unconditional)
6. Wire the session shape (OPENING/CORE/CLOSE) deterministically for
   Library mode instead of relying on an LLM-generated lesson plan
7. Transcribe Deliveries 1–2 into `educational-brain/foundations/` and
   the nine planned libraries — this is the highest-ROI single step
   because it fills the action selector's libraries 2–6 simultaneously

None of these require a new orchestration layer. The orchestrator exists.
The content and rules need to reach it.
