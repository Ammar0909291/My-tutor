# Action Type Audit — `/api/learn/chat` response surface

Branch: `claude/my-tutor-foundation-KDSUO`. Audit date: 2026-06-27.
Scope: every distinct content-block "type" that the AI/route pipeline can produce in or
alongside `msg.content` (`src/app/api/learn/chat/route.ts`), and how each one is consumed
downstream. Read-only — no code changed to produce this document.

There is **no single unified `ContentBlock` type**. Three independent JSON response fields
(`visual`, `visualSpec`, `sceneSpec`) plus five embedded-in-text tags coexist on one
assistant message. Each is catalogued below.

## Confirmed absent from this pipeline
Audio/TTS, flashcards, quizzes (standalone), images, and code-execution blocks are **not**
produced by `/api/learn/chat` at all — each lives in a wholly separate component/route
outside this response shape.

## Gaps

**A) Types missing entirely (never generated by the pipeline)**
- None found within the existing `msg.content`/visual/sceneSpec schema scope — every
  type the system prompt instructs the AI to emit has a corresponding extraction/parsing
  path. (Audio, flashcards, quizzes, images, code-execution are out-of-scope architecturally,
  not "missing" — they're deliberately separate subsystems, not gaps in this response shape.)

**B) Types generated but not rendered with dedicated UI**
- #6 Inline practice MCQ — generated and present in `msg.content`, but rendered as bare
  markdown text with no distinct question/options/answer UI, no answer-capture affordance,
  and no scoring hook (unlike the separate quiz/flashcard components elsewhere in the app).

**C) Types rendered but not bias-weighted in `teachingOutputBias.ts`**
- #1 `visual` (VisualType tag) — rendered via `VisualCard`, never inspected by
  `isOptionalVisual()`/`deriveOutputBias()`. A `MOMENTUM_RECOVERY`/`CONFIDENCE_BUILDING`
  strategy that should suppress optional visuals currently only suppresses `VisualSpec`
  visuals, not `VisualType` ones — an inconsistency in the bias layer's coverage.
- #3, #4, #5 `sceneSpec` (rule-based, parametric, and free-form) — none are inspected by
  the bias module at all, despite being a real, possibly-heavy 3D visual the same
  suppression logic should arguably apply to.

**D) Types not covered by any harness**
- #7 `[ASSESSMENT_RESULT...]` tag parsing/dispatch — no harness exercises the regex
  extraction or the resulting call into `/api/assessment/evaluate`.
- #8 `[MATH_ANSWER]`/`[CODE_ANSWER]` tag parsing — no harness.
- #9 `[LESSON_COMPLETE]` tag parsing/dispatch — no harness.
- #1 and #6 each have harnesses for their *generation/decision* logic but not for the
  end-to-end client-side consumption path (`LessonScreen.tsx` parsing).

## Recommendation for Step 3

The highest-value, lowest-risk gap is **C) the `visual`/`VisualType` and `sceneSpec`
output-bias gap** (item C above): `isOptionalVisual()` already exists and the
`STRATEGY_BIAS` table already exists — the fix is extending bias coverage to the two
content types it currently ignores, with zero schema changes and zero new UI. This is a
real pedagogical gap (a student in `MOMENTUM_RECOVERY` should not be shown an optional 3D
scene or VisualType card any more than an optional `VisualSpec` diagram), it's narrowly
scoped, and it doesn't touch scene generators, `VisualCard`, or narration code — only
`teachingOutputBias.ts` and the call sites in `route.ts` that already exist for the
`VisualSpec` suppression path.

The B) gap (inline-practice UI) would require new "candy design system" UI components and
touches rendering more broadly — larger scope, deferred per the instruction to stop rather
than partially implement if Step 3 scope is too large for a type B/D fix. **Proceeding with
the C) gap as Step 3.**
