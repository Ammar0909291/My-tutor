## 0. Why This Grammar Exists

Without a composition grammar, protocol authors make ad hoc sequencing decisions. The AI fills the gaps. Two authors produce incompatible protocols. The AI Removal Test fails.

With a composition grammar, protocol authoring becomes **generative**: given a concept's KG properties and a student's entry state, the grammar produces a valid Teaching Action sequence. The AI voices the result — it does not construct it.

This grammar operates at **three levels**:

| Level | Unit | Contains |
|---|---|---|
| 1 | Teaching Action (TA) | Primitive sequence — one pedagogical move |
| 2 | Teaching Protocol | TA sequence — one student-state path through a concept |
| 3 | Session | Protocol sequence — one learner's runtime session |

Every rule in this grammar constrains one of these levels. Rules are labelled **GR** (primitive-level), **PA** (protocol-level), **SE** (session-level), or **AIR** (AI Removal invariants).

GR-1 through GR-10 from the Primitive Library are reproduced and formalised here. No rule here contradicts them.

## 1. Grammar Notation

| Symbol | Meaning |
|---|---|
| `→` | Followed by (sequence) |
| `\|` | Or (alternative) |
| `( )` | Grouping |
| `?` | Optional (0 or 1 occurrences) |
| `+` | One or more |
| `*` | Zero or more |
| `×n` | Exactly n repetitions |
| `{m,n}` | Between m and n repetitions |
| `[condition]` | Only if condition holds |
| `¬` | Forbidden in this context |
| `⟹` | Triggers branch (signal-conditional) |
| `⊕` | Simultaneously (parallel primitives) |
| `::=` | Defined as |
| `SIGNAL:X` | A deterministic response classifier output: CORRECT, INCORRECT, MISCONCEPTION, PARTIAL, NO_RESPONSE, STATE_UPGRADE |

All identifiers (P01, S0, CPA:C, etc.) refer to items in the Primitive Library. Content slot values (`[concept_id]`, `[example_instance]`) are filled from the Knowledge Graph at execution time — never generated by AI.

## 2. Level 1 — Teaching Action Grammar

### 2.1 Canonical Teaching Action Structure

```
teaching_action ::=
    opening
    → input_phase?
    → processing_phase?
    → elicitation_cycle+
    → regulation_tail?
```

Every clause is defined below. A valid TA must contain at minimum: `opening → elicitation_cycle`. All other clauses are optional.

### 2.2 Opening

```
opening ::=
    P01                   // New concept | fresh session | attention drift detected
  | P02                   // Continuing topic | prior schema present
  | P02 → P03             // Schema + orientation needed (state change between sessions)
  | P01 → P02             // Full open: attention then schema activation
  | P01 → P04             // Attention then context-embedding (concrete-first protocols)
  | P01 → P05             // Attention then curiosity induction (inquiry-first)
  | P01 → P69             // Attention then relevance anchoring (S8 entry)
  | P01 → P95             // Attention then interpretive frame (English literature concepts)
  | P02 → P70             // Schema + competence evidence (S6 recovery)
  | P02 → P72             // Schema + authority transfer (advanced concepts)
```

**GR-1 (formalised):** The first element of `opening` is always P01 or P02. No exceptions.

A-category primitives P69–P73 and P95 appear only after P01 or P02 — never as the first element.

### 2.3 Input Phase

```
input_phase ::=
    concrete_phase
  | perceptual_phase
  | concrete_phase → perceptual_phase
  | concrete_phase → abstract_phase
  | perceptual_phase → abstract_phase
  | concrete_phase → perceptual_phase → abstract_phase

concrete_phase ::=
    P06                   // Concrete embodiment (physical/tangible)
  | P09                   // Narrative framing (episodic anchor)
  | P10                   // Worked example (complete)
  | P11                   // Partial worked example (with gap)
  | P12                   // Faded worked example [Named Compound — expand per GR-10]
  | P13                   // Expert think-aloud

perceptual_phase ::=
    P07[modality]         // Modality: diagram | graph | table | mapping | timeline | phoneme-grid

abstract_phase ::=
    P08                   // Abstract notation (symbols, formal expressions)
```

**GR-3 (formalised):** `abstract_phase` is blocked unless at least one of `concrete_phase` OR `perceptual_phase` has already executed **for this concept in this session**. The CPA session state machine (§5) tracks this. Attempting to execute P08 without prior P06 or P07 in the same session is an **invalid composition**.

### 2.4 Processing Phase

```
processing_phase ::=
    processing_primitive{1,3}   // GR-7: maximum 3 consecutive without elicitation break

processing_primitive ::=
    P14 | P15 | P16 | P17 | P18 | P19 | P20 | P21 | P22 | P23 | P24 | P25   // Core C-category
  | P92                         // Thought Experiment [Physics specialisation]
  | P93                         // Temporal Sequence Assembly [Phonics specialisation]
  | P94                         // Contextual Inference [Vocabulary specialisation]
```

**GR-7 (formalised):** After 3 consecutive processing primitives, an `elicitation_cycle` is mandatory before any further processing_primitive. The counter resets at each `elicitation_cycle`.

Recommended processing sequences by cognitive purpose:

| Purpose | Sequence |
|---|---|
| Introduce new structure | P14 → P15 → P16 |
| Extract principle from examples | P15 → P20 → P21 |
| Test schema depth | P17 → P18 → P22 |
| Build complexity | P23 → P24 → P25 |
| Physics intuition | P14 → P92 → P15 |
| English text interpretation | P95 → P94 → P16 |

### 2.5 Elicitation Cycle

The elicitation cycle is the **cognitive heart** of every Teaching Action. It is the only phase that requires student output.

```
elicitation_cycle ::=
    productive_struggle_guard?
    → elicitation_primitive
    → P55
    → response_branch

productive_struggle_guard ::= P54   // Required if: first-attempt elicitation AND difficulty ≥ HIGH AND student state is S0 or S6

elicitation_primitive ::=
    P34   // Closed question — specific recall or application
  | P35   // Open question — surfaces student reasoning
  | P36   // Probing question — deepens a prior response
  | P37   // Counterexample question — tests generalisation boundary
  | P38   // Reformulation prompt — forces representation shift
  | P39   // Transfer prompt — initiates cross-context application
  | P40   // Metacognitive prompt — triggers self-monitoring
  | P41   // Diagnostic question — detects specific misconception
  | P42   // Example generation — student produces an example
  | P43   // Non-example generation — student produces counterexample
  | P44   // Definition construction — student builds definition
  | P45   // Problem construction — student creates problem
  | P46   // Procedure construction — student designs procedure
  | P47   // Diagram construction — student builds visual
  | P48   // Analogy construction — student invents analogy
```

**GR-2 (formalised):** `P55` must immediately follow every `elicitation_primitive`. There is no valid composition in which an `elicitation_primitive` fires and the next element is not `P55`. The minimum wait duration is 5 seconds; the maximum before response_branch is 30 seconds.

**GR-8 (formalised):** `P54` precedes any elicitation where: (a) the elicitation is first-attempt for this concept, AND (b) the elicitation type is P35/P42/P44/P45/P46, AND (c) difficulty is HIGH. `P54` may appear before any other elicitation at author discretion.

**Response taxonomy schema (AIR-2 implementation):** Every elicitation primitive in an authored Blueprint must carry a response taxonomy as a structured list, not prose. Minimum schema per entry: `{ stimulus: "<what the student produces>", signal: "<one of 6 signals>", mc_id: "<MC-ID if MISCONCEPTION, else null>" }`. Maximum 8 entries per elicitation primitive. If a student response matches no entry, the classifier returns SIGNAL:PARTIAL. SIGNAL:PARTIAL means the response is directionally correct but incomplete — it contains no identifiable incorrect element. A response containing incorrect elements returns SIGNAL:INCORRECT, not SIGNAL:PARTIAL.

### 2.6 Response Branch

```
response_branch ::=
    SIGNAL:CORRECT      ⟹ success_path
  | SIGNAL:INCORRECT    ⟹ failure_path
  | SIGNAL:MISCONCEPTION ⟹ misconception_path   // Requires prior P41 or P64 signal — GR-4
  | SIGNAL:PARTIAL      ⟹ elaboration_path
  | SIGNAL:NO_RESPONSE  ⟹ wait_extension
  | SIGNAL:STATE_UPGRADE ⟹ protocol_reselect

// ─── Success path ──────────────────────────────────────────────────────────────
success_path ::=
    P49                              // Confirm correct reasoning
    → (P53 | P36 | P39 | P40)?      // Optional: elaborate | probe deeper | transfer | metacognition

// ─── Failure path ──────────────────────────────────────────────────────────────
failure_path ::=
    P50                              // Disconfirm
    → P51                            // Diagnose error (procedural vs. conceptual)
    → (P52 | P65)                    // Redirect attention | Attribute error type
    → (P30 | recovery_sequence)      // Bridge if schema adjacent | Recover if CPA regression needed

// ─── Elaboration path (partial response) ──────────────────────────────────────
// PARTIAL = directionally correct, incomplete, no incorrect element present.
// A response with incorrect elements returns SIGNAL:INCORRECT, not SIGNAL:PARTIAL.
elaboration_path ::=
    P49                              // Acknowledge what is correct (no [partial] qualifier needed)
    → P53                            // Request elaboration
    → P55                            // Wait
    → (success_path | failure_path)  // Re-classify after elaboration

// ─── Wait extension (no response) ─────────────────────────────────────────────
wait_extension ::=
    P55 × 2                          // Additional wait (10 s total)
    → [if still no response] → P52   // Narrow the question
    → elicitation_primitive          // Re-elicit at lower complexity
    → P55

// ─── State upgrade ─────────────────────────────────────────────────────────────
protocol_reselect ::=
    [update state classification]
    → [select new Protocol entry point]
    → [resume from matching TA]
```

**GR-4 (formalised):** `misconception_path` is a gated branch. Entry requires a SIGNAL:MISCONCEPTION output from either P41 (Diagnostic Question) or P64 (Confidence Calibration). Entering `misconception_path` without this gate signal is an invalid composition.

### 2.7 Regulation Tail

```
regulation_tail ::=
    metacognitive_close?
    → scaffolding_adjustment?
    → pacing_adjustment?
    → modality_adjustment?

metacognitive_close ::= P63 | P64 | P65 | P66 | P67 | P68
scaffolding_adjustment ::= P81 | P82
pacing_adjustment ::= P84 | P85
modality_adjustment ::= P86
```

The regulation tail is optional. It appears only when the TA's elicitation result signals that cognitive state management is needed before the next TA begins.

## 3. Level 2 — Protocol Grammar

### 3.1 Protocol Structure

```
teaching_protocol ::=
    [student_state: S0–S9 (may combine)]
    diagnostic_ta
    → concept_ta+
    → consolidation_ta?
    → assessment_ta
    → [if mastery] → mastery_close
    → [if not mastery] → recovery_sequence

diagnostic_ta ::=
    (P01 | P02)
    → P41
    → P55
    → response_branch           // State classification from response
    → [route to matching Protocol entry]

concept_ta ::=
    teaching_action             // One pedagogical move
    → [output signal routes to next concept_ta | consolidation_ta | assessment_ta | recovery]

consolidation_ta ::=
    opening
    → (P58 | P59 | P60 | P61)   // Elaborative interrogation | self-explanation | variation | summary
    → P55
    → response_branch

assessment_ta ::=
    [position: not first in session unless session type = DIAGNOSTIC — GR-9]
    opening
    → assessment_sequence
    → response_branch

assessment_sequence ::=
    probe{2,5}                  // Minimum 2; maximum 5 without consolidation break

probe ::=
    (P74 | P75 | P76 | P77 | P78 | P79 | P80)
    → P55

mastery_close ::=
    P91                         // Mastery Verification — terminal; nothing follows — GR-6
    → P68                       // Mastery Self-Declaration
    → P62                       // Spaced Review Scheduling

recovery_sequence ::=
    [see §6]
```

**PA-1:** The first TA in any Protocol is always a `diagnostic_ta` unless student state is pre-known from a prior session record.

**PA-2:** Every TA in a Protocol outputs one of six signals: SUCCESS, FAILURE, MISCONCEPTION, PARTIAL, NO_RESPONSE, STATE_UPGRADE. The signal determines which TA follows.

**PA-3:** Maximum 7 TAs per session before P62 fires. If mastery has not been reached at TA-7, schedule P62 and close the session. Do not push more content.

**PA-4:** Every TA must contain at least one `elicitation_primitive`. There are no passive-only TAs. The student is always asked to produce output.

**PA-5:** P90 (FORMATIVE ASSESSMENT WITH FEEDBACK — expanded form) must appear in at least one TA before mastery can be declared.

**PA-6:** P91 terminates the Protocol's teaching sequence for the concept. Nothing in the Grammar may follow P91 except P68 and P62.

### 3.2 Protocol Assembly for State Combinations

Students can hold multiple states simultaneously. Combination rules:

| Primary State | Secondary State | Resolution |
|---|---|---|
| S1 (Procedural) | S7 (Overconfident) | S7 takes priority — use counterexample-first before any input phase |
| S2 (Misconception) | S6 (Low Confidence) | S2 route but replace P28 with P30 throughout — GR-5 |
| S2 (Misconception) | S7 (Overconfident) | S2 route with P27 mandatory before P28 — S7 needs explicit exposure |
| S0 (Novice) | S9 (Second Language) | Maximise P06/P07; minimise P08; narrate P07 with P13; defer P44/P46 |
| S4 (Prereq Gap) | S6 (Low Confidence) | Prerequisite repair first; use P70 at every success to build confidence |
| S5 (Mature) | S8 (Adult) | Begin at abstract_phase; use P39 frequently; P06 feels condescending — omit |
| Any | S6 | P71 fires once at session open; P54 precedes all P35/P42 elicitations |

## 4. CPA Session State Machine

The CPA stage is a session-scoped tracker per concept. It starts at `C` for any concept that has never been taught to this student.

```
CPA_stage ∈ { C, P, A }

initial_stage(concept_id, student_id):
    if no prior session record for (concept_id, student_id) → C
    else → recorded_stage from StudentMemory

transition_rules:
    C → P: P07 has fired AND P15 or P16 returned SIGNAL:CORRECT after P06
    P → A: P08 has fired AND P15 or P16 returned SIGNAL:CORRECT after P07
    A → A: (stays; all abstract work proceeds in A)
    A → P: [regression] failure_path traversed twice in A-stage without success
    P → C: [regression] failure_path traversed twice in P-stage without success
    C → C: [stays; simplify concrete instance]
```

GR-3 is enforced by this machine: `abstract_phase` may only fire when `CPA_stage = P` or `CPA_stage = A`. The machine blocks P08 when `CPA_stage = C`.

At session end: persist current `CPA_stage` to StudentMemory. Next session resumes from stored stage.

## 5. Branching Grammar — Complete Decision Table

The Response Classifier outputs one signal after every `P55`. The Grammar's branching table maps (signal × student_state × CPA_stage) → next_element:

| Signal | Student State | CPA Stage | Next Element |
|---|---|---|---|
| CORRECT | Any | Any | success_path |
| CORRECT | Any | C or P | [if protocol calls for it] CPA_advance → next concept_ta |
| INCORRECT | Any | A | failure_path → CPA_regression to P |
| INCORRECT | Any | P | failure_path → CPA_regression to C |
| INCORRECT | Any | C | failure_path → simplified concrete |
| MISCONCEPTION | S2 or S7 | Any | misconception_path (Schema Repair — §7) |
| MISCONCEPTION | S6 | Any | P30 (Bridge) — skip P28 per GR-5 |
| PARTIAL | Any | Any | elaboration_path |
| NO_RESPONSE | S6 | Any | wait_extension → P52 with simpler form |
| NO_RESPONSE | S0 | Any | wait_extension → P06 (return to concrete) |
| NO_RESPONSE | Other | Any | wait_extension → re-elicit |
| STATE_UPGRADE | S0→S3 | Any | protocol_reselect → Protocol E entry |
| STATE_UPGRADE | S1→S3 | Any | protocol_reselect → Protocol E entry |
| STATE_UPGRADE | S3→mastery | Any | assessment_ta |

## 6. Misconception Repair Grammar

Entry gate: SIGNAL:MISCONCEPTION must have been returned by P41 or P64. Entering this chain without the gate signal is invalid (GR-4).

```
misconception_repair_chain ::=
    P26                          // Surface the existing mental model explicitly
    → P27?                       // Optionally: articulate schema — mandatory for S7; optional for S2
    → conflict_gate              // Choose path based on S6 check (GR-5)
    → [branch]

conflict_gate [if NOT S6]:
    P28                          // Cognitive Conflict Induction — present contradicting evidence
    → P29                        // Conflict Resolution Pause — hold dissonance; do not resolve yet
    → P30                        // Bridge Construction — connect old schema to correct one
    → P31                        // Schema Replacement — install correct schema
    → P32                        // Schema Consolidation — stabilise new understanding
    → P33?                       // Discrimination Training — distinguish from adjacent schemas
    → resume_point

conflict_gate [if S6]:
    ¬P28                         // GR-5: NEVER conflict-induce a low-confidence student
    P30                          // Bridge directly — connect to something the student does understand
    → P31
    → P32
    → P33?
    → P70                        // Competence Evidence — rebuild confidence after repair
    → resume_point

resume_point ::=
    [return to the interrupted protocol at the correct CPA stage]
    → [re-elicit on the same item using a simpler form]
    → [if correct] → continue protocol
    → [if incorrect again] → recovery_sequence
```

Misconception catalogue integration: The Knowledge Graph asset for each concept carries a named misconception list (MC1…MCn). Each named misconception maps to a specific `P26` content slot — the specific mental model to surface — and a specific `P28` content slot — the specific contradicting evidence. These are not generated by AI; they are authored in the concept's Teaching Protocol.

**Multiple Active Misconception Rule (MAMR):** When `active_misconceptions[]` contains more than one entry, apply these rules in order: (1) if one MC is foundational to another (repairing it would make the other MC's evidence invalid or redundant), repair the foundational one first; (2) if no precedence relationship exists, repair in detection order (FIFO — first detected, first repaired); (3) complete one full repair chain through P32 before entering the next; (4) each repair chain counts as one TA against the PA-3 session cap. The Blueprint's Misconception Engine (Component 6) may optionally declare a `foundational_for: [MC-ID]` field on any MC to make the precedence relationship explicit rather than author-inferred.

## 7. Assessment Grammar

Assessment primitives (P74–P80) test **schema quality**, not answer correctness. They are ordered by cognitive demand:

| Probe | Bloom Level | What it tests |
|---|---|---|
| P77 GENERATION | Create | Can student produce a new valid instance? |
| P76 TRANSFER | Analyse | Can student apply schema in an unfamiliar context? |
| P75 BOUNDARY | Evaluate | Does student know where the schema stops being valid? |
| P74 CLASSIFICATION | Apply | Can student apply schema as a decision criterion? |
| P78 EXPLANATION | Understand | Can student narrate schema structure? |
| P79 PREDICTION | Apply | Can student apply schema predictively? |
| P80 DUAL-TASK | Analyse | Can student correctly classify both positive and negative cases? |

### 7.1 Probe Ordering Rule

For maximum diagnostic information, deploy probes from highest to lowest Bloom level. This catches over-learned surface knowledge (a student can classify but not generate).

**Recommended sequence for a standard assessment_ta:**

```
assessment_ta_standard ::=
    P02                   // Activate
    → P77 → P55           // Generation first (highest Bloom)
    → response_branch
    → P76 → P55           // Transfer second
    → response_branch
    → P74 → P55           // Classification (confirming the standard criterion)
    → response_branch
    → [if all correct] → P90 → P91  // Formative then mastery gate
```

### 7.2 Formative Assessment (P90) Grammar

P90 is a Named Compound. Expand per GR-10 before composition validation:

```
P90_expansion ::=
    [P74 | P75 | P76 | P77 | P78 | P79 | P80]   // Select probe based on protocol position
    → P55
    → [P49 | P50]                                 // Confirm or disconfirm
    → P51                                         // Diagnose error (always — even on confirm, to surface reasoning)
    → P35                                         // Open question: "walk me through your thinking"
    → P55
```

### 7.3 Mastery Verification (P91) Grammar

P91 is the terminal compound (GR-6). Expand before composition validation:

```
P91_expansion ::=
    P77 → P55             // Generation probe — must pass
    → P76 → P55           // Transfer probe — must pass
    → P75 → P55           // Boundary probe — must pass
    → P74 → P55           // Classification probe — must pass
    → P78                 // Explanation probe — terminal element in P91
    → P55                 // Wait
    // GR-6: nothing follows this except P68 and P62
```

Mastery declared only if ALL five probes return SIGNAL:CORRECT. A single INCORRECT or PARTIAL on any probe blocks mastery.

## 8. Retrieval Grammar

### 8.1 Spaced Retrieval (P56)

P56 fires only when sufficient time has elapsed since last successful encoding. The Grammar enforces this via a session-position check:

```
retrieval_ta ::=
    [session_position: NOT the first session for this concept with this student]
    P02                    // Activate — "last time we covered..."
    → P56[time_gap]        // Spaced retrieval — gap must be specified in content slot
    → P55
    → response_branch
```

`P56` is forbidden in the student's first session on a concept. There is nothing to retrieve.

### 8.2 Spaced Repetition Schedule (P89)

P89 is a Named Compound. Expand per GR-10:

```
P89_expansion ::=
    P56[gap: 1 day]       // Interval 1
    → P56[gap: 3 days]    // Interval 2 — fires only if interval 1 returned SIGNAL:CORRECT
    → P56[gap: 7 days]    // Interval 3
    → P56[gap: 21 days]   // Interval 4
    → P56[gap: 60 days]   // Interval 5
```

Each interval fires only after the prior P56 returned SIGNAL:CORRECT. A SIGNAL:INCORRECT at any interval resets the schedule to interval 1.

### 8.3 Interleaved Practice (P57)

P57 is a Named Compound. Expand per GR-10:

```
P57_expansion ::=
    P88                   // Retrieval practice (test conditions — no scaffolding)
    ⊕ P83                 // Interleaving control (deliberate mixed-concept interference)
```

**Invalid within P57:** P81 (Scaffolding) may not appear in the same TA as P88. Retrieval practice requires no scaffolding — adding P81 converts retrieval into guided practice and defeats the retrieval benefit.

### 8.4 Retrieval Practice (P88)

```
retrieval_practice_ta ::=
    P02
    → ¬P10 ¬P11 ¬P12 ¬P81   // No worked examples; no scaffolding
    → elicitation_primitive    // Direct recall under test conditions
    → P55
    → response_branch
```

## 9. Recovery Grammar

Entry condition: `failure_path` has been traversed twice for the same item without a SIGNAL:CORRECT.

```
recovery_sequence ::=
    CPA_regression                      // Step back one CPA stage
    → P85                               // Slow pacing — reduce information rate
    → P84                               // Load partition — isolate one sub-component
    → new_input_phase[prior_CPA_stage]  // Re-enter input at the recovered stage
    → simplified_elicitation            // Lower Bloom level; simpler form
    → P55
    → [if CORRECT] → CPA_advance_slowly → resume_protocol
    → [if INCORRECT] → prerequisite_check

prerequisite_check ::=
    P41[prerequisite_item]    // Probe whether a prerequisite is missing (S4 detection)
    → P55
    → [if missing prereq confirmed] → flag S4 → suspend concept → schedule prereq session
    → [if prereq present] → P65[error attribution] → P52[redirect] → re-elicit
```

CPA regression specifics:

| From Stage | Regress To | First Element After Regression |
|---|---|---|
| A (abstract) | P (perceptual) | P07[same concept, different modality] |
| P (perceptual) | C (concrete) | P06[simpler physical instance] |
| C (concrete) | C (simplified) | P06[minimal example, single variable] |

## 10. Mastery Grammar

Mastery declaration requires all five conditions to be true simultaneously:

| Condition | Primitive Evidence |
|---|---|
| M1: Formative assessment passed | P90 has fired; all probes CORRECT |
| M2: Mastery gate passed | P91 has fired; all 5 probes CORRECT |
| M3: Abstract stage reached | CPA_stage = A for this concept |
| M4: No active misconception | Schema Repair chain completed if any MC triggered |
| M5: Confidence-accuracy calibration | P64 fired; student confidence matches actual performance |

```
mastery_declaration ::=
    [if M1 ∧ M2 ∧ M3 ∧ M4 ∧ M5]
    → P68             // Mastery Self-Declaration — student claims mastery
    → P62             // Spaced Review Scheduling — set next retrieval date
    → [unlock downstream KG nodes — signal to Knowledge Graph engine]
    → [session close]
    
    [if ¬M5 only: overconfident student] → ¬mastery → P37 → P55 → re-assess
    [if ¬M2: missed a probe] → ¬mastery → P52[target the failed probe type] → additional_concept_ta
    [if ¬M3: abstract not reached] → ¬mastery → push abstract_phase → P08 → concept_ta
    [if ¬M4: active misconception] → ¬mastery → misconception_repair_chain
```

A student who passes M2 (all five probes) but fails M5 (calibration — overconfident on incorrect items) must NOT be declared master. This is the S7 trap. Protocol must continue.

## 11. Invalid Primitive Combinations

Beyond GR-1 through GR-10. All of these are hard rejects:

| Code | Invalid Combination | Reason |
|---|---|---|
| IC-1 | P08 without prior P06 or P07 this session | GR-3 — CPA machine blocks this |
| IC-2 | Elicitation primitive without immediate P55 | GR-2 — wait time is non-negotiable |
| IC-3 | P28 for S6 student | GR-5 — conflict induction harms low-confidence learners |
| IC-4 | P26–P32 without P41/P64 MISCONCEPTION signal | GR-4 — schema repair gated |
| IC-5 | Any primitive after P91 in same TA except P68 and P62 | GR-6 — P91 is terminal |
| IC-6 | P74–P80 as first TA in non-diagnostic session | GR-9 — assess after teaching |
| IC-7 | Four+ consecutive C-category without E break | GR-7 — processing load limit |
| IC-8 | Named Compound name used without expansion | GR-10 — validate expanded form only |
| IC-9 | P88 (Retrieval) with P81 (Scaffolding) in same TA | Scaffolding defeats retrieval conditions |
| IC-10 | P49 and P50 in same TA | Contradictory feedback — cannot confirm and disconfirm simultaneously |
| IC-11 | P62 before P91 | Cannot schedule retention before mastery gate is passed |
| IC-12 | P56 (Spaced Retrieval) in student's first session on concept | Nothing to retrieve |
| IC-13 | P68 (Mastery Self-Declaration) without prior P91 | Self-declaration requires probe evidence |
| IC-14 | P71 (Peer Comparison Removal) for non-S6 students | S6-specific intervention — irrelevant otherwise |
| IC-15 | P83 (Interleaving Control) with P84 (Load Management) in same TA | Interleaving adds load; load management reduces it — directly contradictory |
| IC-16 | P01 in the middle of a TA body (not opening) | P01 is an opening primitive — mid-TA attention reset requires session break first |
| IC-17 | P66 (Strategy Awareness) without a prior processing_phase | Cannot name a strategy that hasn't been used |
| IC-18 | P67 (Transfer Readiness Check) before P76 (Transfer Probe) | Self-assessment precedes the probe — check readiness, then test it |
| IC-19 | P54 after P34 (Closed Question) | P54 is productive struggle permission — closed questions are not struggle-inducing |
| IC-20 | P70 (Competence Evidence) as opening primitive | P70 requires prior successful interactions as evidence — not valid on first turn |

## 12. Minimal Complete Teaching Protocol

### 12.1 Minimal Teaching Action

The irreducible TA — remove any element and the grammar breaks:

```
minimal_ta ::= P02 → P34 → P55 → (P49 | P50)
```

- `P02`: required by GR-1
- `P34`: required — at least one elicitation_primitive per TA (PA-4)
- `P55`: required by GR-2
- `P49 | P50`: required — response to student output (failure to respond is a null TA)

This is 4 primitives. It is the floor. No valid TA has fewer than 4 primitives.

### 12.2 Minimal Complete Protocol

The minimum valid three-TA protocol:

```
minimal_protocol ::=
    [TA-1: Diagnostic]
        P01
        → P41             // Diagnostic question — classify student state
        → P55
        → response_branch → [state = S0: continue | other: re-route]

    → [TA-2: Teach]
        P01
        → P06             // Concrete embodiment
        → P34             // Closed question on the concrete
        → P55
        → (P49 | P50)

    → [TA-3: Assess]
        P02
        → P74             // Classification probe
        → P55
        → response_branch
        → P91             // Mastery gate [if CORRECT after remediation]
```

This is the minimum: diagnose → teach → assess. Any protocol shorter than 3 TAs cannot satisfy PA-1 (diagnostic), PA-4 (elicitation in each TA), and GR-9 (assessment not first) simultaneously.

## 13. Complex Protocol Construction

### 13.1 Composition Order

Build complex protocols in this order:

1. **Identify entry state** (S0–S9, may combine)  
2. **Select CPA entry stage** (C for S0/S9/S4; P for S3; A for S5)  
3. **Build diagnostic_ta** (always first)  
4. **Choose input_phase** based on CPA entry stage  
5. **Select processing_phase** for the concept's key cognitive structure  
6. **Design elicitation_cycle** — select primitive by what claim you're testing  
7. **Write response_branch** — both success and failure paths must be fully specified  
8. **Add regulation_tail** only if cognitive state management is needed  
9. **Place P90** — one formative assessment cycle before final gate  
10. **Terminate with P91** — five-probe mastery gate  
11. **Close with P68 → P62** if mastery declared  
12. **Validate against IC-1 through IC-20** — reject any invalid combination  
13. **Apply AI Removal Test** — verify each element passes all four checks (§15)

### 13.2 Multi-State Protocol

When a student holds two states (e.g., S1+S7):

```
multi_state_protocol(S1, S7):
    [TA-1] diagnostic_ta → confirm S1+S7
    [TA-2] P02 → P37[counterexample: non-formula function] → P55 → response_branch
           // S7 needs a challenge — counterexample first, before any confirmation
    [TA-3] [if resists: S7 confirmed] → P41[targeted: "what makes this fail?"] → P55 → response_branch
    [TA-4] [misconception_path: formula-only schema]
           P26 → P27 → P28[present lookup table with no formula] → P29 → P30 → P31 → P32
    [TA-5] P02 → P42[generate a non-formula function] → P55 → response_branch
    [TA-6] assessment_ta → P90 → P91
```

### 13.3 Branching Protocol Map

Every Protocol is a directed graph where:
- **Nodes** are Teaching Actions
- **Edges** are labelled with response signals (CORRECT, INCORRECT, MISCONCEPTION, PARTIAL, STATE_UPGRADE)
- **Leaves** are one of: mastery_close | recovery_cap (max TAs reached) | suspension (prereq gap)

Every node must have at least two outgoing edges (the grammar is always branching). A Protocol with any node that has only one outgoing edge is under-specified.

### 14.2 Protocol G — S1 (Procedural Carrier — formula-only schema)

```
Concept: math.func.function-concept
Entry state: S1
CPA entry: P (perceptual — student has seen graphs; exploit existing representation)

[TA-1: Diagnostic]
    P01
    → P41[content: "Is this table a function? Input 1→4, Input 2→7, Input 3→10"]
    → P55

[Schema Repair — MC: formula-required misconception]
    P26[surface: "You said you need a formula — what does a formula guarantee?"]
    → P27[explicit: "Your criterion is: must have formula. Let's test that criterion."]
    → P28[conflict: "Here is f(x) defined by a lookup table: 4 cities → 4 populations.
                     Is this not a function because it has no formula?"]
    → P29[hold: "Take a moment. The table works — does it need a formula?"]
    → P30[bridge: "What a formula does: guarantee each input maps to one output.
                   The table does the same thing without a formula."]
    → P31[replace: "A function is any assignment where each input has exactly one output.
                    Formula-or-not is irrelevant."]
    → P32[consolidate: "The rule is single-valuedness. The formula is one way to express it."]
    → P33[discriminate: "Formula → function (usually). Function → formula? NOT always."]

[TA-2: Resume at Perceptual]
    P02
    → P07[modality: three representations — table, graph, arrow diagram — same function]
    → P16[compare: "What do all three have in common?"]
    → P35[content: "What is the property that makes all three the same function?"]
    → P55
    → success_path → P49 → P39[transfer: "Apply this to a pricing table at a supermarket"]
    → P55

[TA-3: Assessment]
    P02
    → assessment_sequence:
        P77[generate a function without a formula] → P55 → response_branch
        → P75[boundary: sideways parabola] → P55 → response_branch
        → P74[classify a set of ordered pairs with repeated y-values] → P55 → response_branch

    → P91_expansion

```

### 14.3 Schema Repair — S2+S6 (Misconception Carrier + Low Confidence)

```
Concept: math.func.function-concept
Misconception: "Vertical Line Test is the definition of a function" (MC-VLT)
Entry states: S2 (misconception present) + S6 (low confidence)

[TA-1: Diagnostic]
    P01
    → P71[S6 intervention: "There are no right answers here — just thinking out loud"]

    → P55

[Schema Repair — S2+S6 path: ¬P28 per GR-5]
    P26[surface: "You used the Vertical Line Test — tell me what that test checks"]
    → P27[explicit: "So your definition of function = passes VLT"]
    // ¬P28 — do not induce conflict in S6
    P30[bridge: "The VLT works because of something deeper: single-valuedness.
                 Let's check: if x = 1 in x = y², what are the possible y values?"]

                    One input, two outputs. That violates single-valuedness.
                    The VLT would show this too — but the reason isn't the VLT.
                    The reason is two outputs."]
    → P32[consolidate: "Test to use: does any input have more than one output? If yes → not a function."]

    P02
    → P07[modality: two graphs side by side — parabola (function) vs. sideways parabola (not)]

    → P35[content: "What is different about these two graphs?"]
    → P55
    → success_path → P49 → P53[elaborate: "Say more about why that difference matters"]
    → P55

    P02
    → P64[calibrate: "How confident are you now that you could classify any relation?"]
    → assessment_sequence[3 probes: P74, P75, P78]

    → [if M1∧M2∧M3∧M4∧M5] → P91 → P68 → P62
```

## 15. AI Removal Test Invariants

Every primitive sequence produced by this grammar must pass all four AIR invariants. These are checked at authoring time — not runtime.

### AIR-1 — Finite Input Values

Every `elicitation_primitive` and `processing_primitive` has a content slot with a finite, enumerable set of valid values drawn from the Knowledge Graph. No primitive may have an input whose valid values are infinite or require AI to generate.

| Primitive | Content Slot | Source |
|---|---|---|
| P06 | `concrete_instance` | KG asset: authored concrete examples list |
| P07 | `modality`, `diagram_id` | KG asset: authored visual list |
| P34 | `question_text`, `expected_answers[]` | KG asset: authored closed questions |
| P41 | `diagnostic_text`, `misconception_ids[]` | KG asset: authored misconception list |
| P28 | `conflict_evidence` | KG asset: authored conflict evidence per misconception |
| P74–P80 | `probe_item` | KG asset: authored probe items |

If a content slot requires AI to generate its value, the asset is incomplete and must be returned to the author before use.

### AIR-2 — Response Taxonomy Per Elicitation

Every `elicitation_primitive` has a pre-authored response taxonomy: a finite set of response categories that the Response Classifier can output. The taxonomy is encoded in the concept's Teaching Protocol — never derived by AI at runtime.

Response categories for `P34` (Closed Question): { CORRECT, INCORRECT, PARTIAL }  
Response categories for `P35` (Open Question): { CORRECT, INCORRECT, PARTIAL, MISCONCEPTION, PARTIAL_MISCONCEPTION }  
Response categories for `P41` (Diagnostic Question): { MISCONCEPTION:MC-ID, NO_MISCONCEPTION, AMBIGUOUS }  
Response categories for `P74–P80` (Probes): { CORRECT, INCORRECT, PARTIAL }

The Response Classifier maps a student's natural language response to one of these categories. It does not interpret meaning — it matches against the pre-authored taxonomy.

### AIR-3 — Deterministic Branching

The `response_branch` is deterministic: given (SIGNAL, student_state, CPA_stage), exactly one path is selected. There is no path that requires the AI to "decide what to do next." Every branch in §5's decision table has exactly one output.

If an author writes a branch that requires AI judgment — "if the student seems confused, try..." — that branch is invalid. The author must convert the judgment criterion into a deterministic signal and add it to the response taxonomy.

### AIR-4 — Subject Independence

No primitive in a composition may reference a concept, domain, subject, or student name directly. All references are content slot identifiers that the KG fills at execution time.

Test: Replace every content slot value with `[SLOT]`. The primitive sequence must remain structurally valid and cognitively coherent. If the sequence breaks without a specific concept's content, the primitive itself is under-specified or the composition contains an implicit dependency on a concept.

Example (valid): `P06[concrete_instance: [SLOT]] → P34[question: [SLOT]] → P55 → P49`  
Example (invalid): `P06[concrete_instance: "vending machine"] → P34[question: "What does button A give?"] → ...`  
The invalid example has content fused into the composition. It fails AIR-4.

### AIR-5 — Sequence Fixed at Authoring

The primitive sequence within a TA is determined by the protocol author, not by the runtime AI. The runtime AI (Response Classifier / Voice Renderer) receives a pre-specified sequence, fills content slots from the KG, and renders it. The AI does not reorder, add, or remove primitives.

The boundary:
- **Author decides:** which primitives, in which order, for which student state
- **Engine decides:** which TA to select from the authored set, based on current student state  
- **AI decides:** nothing structural — only how to voice what the engine has selected

## 16. Level 3 — Session Grammar

### 16.1 Session Structure

```
session ::=
    session_open
    → protocol_execution+
    → session_close

session_open ::=
    [retrieve StudentMemory for this student × concept pair]
    → [restore CPA_stage from prior session]
    → [restore active misconceptions]
    → [restore scheduled retrievals (P89)]
    → diagnostic_ta [if first session | state stale]
    → state_classification

protocol_execution ::=
    [select Protocol for current student_state]
    → teaching_action+
    → [update StudentMemory after each TA]

session_close ::=
    [if mastery declared] → P68 → P62 → unlock_downstream_nodes
    [if not mastery] → P62[schedule next session] → [persist CPA_stage + misconceptions + retrieval schedule]
```

**SE-1:** Student Memory is updated after **every** TA — not at session end. A session that fails mid-way (connectivity, timeout) must leave the student in a valid state, not an undefined one.

**SE-2:** The session must not leave the student mid-schema-repair. If `misconception_path` has been entered, the chain must complete to at least P31 before the session can close. If time expires, persist the repair state and resume at P31 in the next session.

### 16.2 Session State at Hand-off

At every session close, persist to StudentMemory:

| Field | Source |
|---|---|
| `cpa_stage` | CPA state machine final stage |
| `active_misconceptions[]` | Any MC signals not yet resolved through P32 |
| `mastery_status` | { NOT_STARTED, IN_PROGRESS, MASTERED } |
| `next_retrieval_date` | From P62 content slot |
| `last_probe_results` | P74–P80 signal outcomes from most recent assessment_ta |
| `confidence_calibration` | Most recent P64 output |
| `protocol_position` | TA index within current Protocol at close |
